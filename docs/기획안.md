**PART 1. 기획서**

1\. 프로젝트 개요

1.1 배경 및 문제 정의

현대 연구 환경에서는 매일 수천 편의 논문이 arXiv, Semantic Scholar,
PubMed 등에 업로드된다. 그러나 연구자 및 개발 조직은 다음과 같은 구조적
병목을 겪고 있다.

-   논문 인지 지연: 최신 논문이 등장해도 실무자가 인지하기까지 평균 수
    주\~수 개월이 소요됨

-   언어 장벽: 영문 논문이 대다수로, 국내 팀의 이해와 활용까지 추가적인
    번역 비용 발생

-   연관성 파악 부재: 논문이 현재 진행 중인 사내 프로젝트·기획안과
    어떻게 연결되는지 수동으로 분석해야 함

-   신뢰도 판단 어려움: 논문의 인용 수, 저널 신뢰도, 방법론 타당성 등을
    개별적으로 검증해야 함

1.2 프로젝트 목표

본 프로젝트는 논문 발표 이벤트를 자동으로 감지하고, 분석·평가·요약까지
자율적으로 처리하는 AI Agent 파이프라인을 구축한다. 궁극적으로는 \'논문
발표 → 사내 보고서 자동 생성\'의 제로터치(Zero-Touch) 워크플로우를
실현한다.

  ------------- --------------------------- ------------------------------
  **구분**      **목표**                    **성공 지표**

  모니터링      신규 논문 자동 감지         감지 지연 \< 1시간

  분석·번역     한국어 요약 자동 생성       요약 품질 MOS ≥ 4.0/5.0

  신뢰도 평가   논문 신뢰 점수 자동 산출    전문가 평가와 상관계수 ≥ 0.8

  연관 분석     사내 문서와의 유사도 매핑   Precision@5 ≥ 0.7

  보고서        프로젝트 연계 보고서 자동   사용자 만족도 ≥ 4.0/5.0
                생성                        
  ------------- --------------------------- ------------------------------

1.3 범위 (Scope)

**In Scope (이번 프로젝트에 포함)**

-   arXiv, Semantic Scholar 대상 논문 모니터링 및 크롤링

-   Kafka 기반 이벤트 스트리밍 파이프라인

-   LLM 기반 논문 요약 및 한국어 번역

-   논문 신뢰도 자동 평가 (인용 수, 저널 IF, 방법론 평가)

-   사내 문서 임베딩 및 유사도 기반 연관 분석

-   최종 조사 보고서 자동 생성 및 대시보드 제공

**Out of Scope (이번 프로젝트에서 제외)**

-   실시간 논문 공동 작성 기능

-   특허 문서 분석

-   외부 사용자 공개 서비스

**PART 2. 시스템 설계서**

5\. 시스템 아키텍처

5.1 전체 흐름 요약

시스템은 크게 6개 레이어로 구성된다. 각 레이어는 독립적으로 배포 및
스케일링 가능하며, Kafka를 중심으로 비동기 이벤트 기반으로 연결된다.

1.  외부 소스 (arXiv, Semantic Scholar) → 논문 모니터링 에이전트가 주기
    크롤링

2.  신규 논문 감지 시 Kafka 토픽 new_paper_detected에 이벤트 발행

3.  3개 에이전트(분석, 신뢰도 평가, 임베딩 생성)가 병렬 구독 및 처리

4.  처리 결과를 각각 Vector DB (논문 임베딩, 사내 문서 임베딩)에 저장

5.  유사도 검색 엔진이 논문 ↔ 사내 프로젝트 연관 관계를 산출

6.  보고서 생성 에이전트가 최종 조사 보고서를 생성 → 사용자 대시보드
    제공

6\. 컴포넌트 상세 설계

6.1 논문 모니터링 에이전트

역할: 외부 학술 데이터베이스를 주기적으로 크롤링하여 신규 논문을
감지하고 Kafka에 이벤트를 발행한다.

-   크롤링 주기: 1시간 (기본값, 설정 가능)

-   지원 소스: arXiv (카테고리 필터링), Semantic Scholar (키워드 검색)

-   중복 감지: 논문 ID(DOI, arXiv ID) 기반 Redis Set으로 중복 방지

-   Kafka 이벤트 스키마: { paper_id, title, authors, abstract, source,
    published_at, url }

6.2 논문 분석 에이전트

역할: Kafka 구독 후 논문 원문을 로드하여 LLM 기반 요약·번역·키워드
추출을 수행한다.

-   입력: new_paper_detected 이벤트 (abstract + 본문 URL)

-   처리: LLM 프롬프트로 한국어 요약(200\~400자), 핵심 기여(3\~5개),
    키워드 추출

-   출력: paper_analyzed 이벤트 발행 + PostgreSQL 저장

-   오류 처리: 본문 로드 실패 시 abstract만으로 요약 (폴백 전략)

6.3 신뢰도 평가 에이전트

역할: 논문의 학문적 신뢰도를 다차원 평가하여 신뢰 점수를 산출한다.

  ------------------ ------------ -----------------------------------------
  **평가 항목**      **가중치**   **측정 방법**

  인용 수 (Citation  25%          Semantic Scholar API 조회, 로그 스케일
  Count)                          정규화

  저널 임팩트 팩터   25%          arXiv 카테고리 매핑, Scimago 저널 순위

  방법론 명확성      30%          LLM 평가: 실험 설계, 데이터셋 명시 여부,
                                  비교 실험

  재현성 지표        20%          코드/데이터 공개 여부, 구현 세부 사항
                                  충분성
  ------------------ ------------ -----------------------------------------

6.4 임베딩 생성 에이전트

역할: 논문 및 사내 문서를 벡터화하여 Vector DB에 저장한다.

-   논문 임베딩: 제목 + 초록 + 키워드를 결합한 텍스트 임베딩 (1536차원)

-   사내 문서 임베딩: 기획안, 프로젝트 보고서를 청크 분할 후 임베딩

-   청크 전략: 512 토큰 슬라이딩 윈도우, 50 토큰 오버랩

-   Vector DB: pgvector (HNSW 인덱스, cosine 유사도)

6.5 유사도 검색 엔진

역할: 신규 논문과 사내 문서 간의 의미론적 유사도를 산출하여 연관
프로젝트를 매핑한다.

-   검색 방식: ANN (Approximate Nearest Neighbor) - HNSW 알고리즘

-   임계값: cosine similarity ≥ 0.70 (연관 있음), ≥ 0.85 (강한 연관)

-   출력: 상위 5개 연관 사내 프로젝트 + 유사도 점수

-   주기: 논문 분석 완료 이벤트 수신 시 즉시 실행

6.6 보고서 생성 에이전트

역할: 분석 결과를 종합하여 최종 조사 보고서를 생성한다.

**보고서 구성 항목:**

-   논문 기본 정보 (제목, 저자, 게재일, 출처 링크)

-   한국어 요약 및 핵심 기여

-   신뢰도 점수 및 평가 근거

-   연관 사내 프로젝트 목록 및 유사도

-   활용 가능성 분석 (LLM이 연관 프로젝트 컨텍스트를 참조하여 생성)

-   추천 액션 아이템 (논문 딥다이브, 실험 적용, 팀 공유 등)

8\. 비기능 요구사항

8.1 성능

-   논문 감지에서 보고서 생성까지 end-to-end 처리 시간 \< 5분

-   Vector DB 유사도 검색 응답 \< 500ms (p99)

-   동시 처리 가능 논문 수: 에이전트당 10개 (Kafka consumer group 수평
    확장)

8.2 가용성 및 신뢰성

-   Kafka 이벤트 at-least-once 보장, 멱등성 처리로 중복 방지

-   각 에이전트 장애 시 자동 재시도 (exponential backoff, max 3회)

-   Dead Letter Queue (DLQ) 로 처리 실패 이벤트 보존 및 수동 재처리

8.3 보안

-   LLM API Key, DB 접속 정보는 AWS Secrets Manager 관리

-   사내 문서 접근 권한: 사용자 역할 기반 접근 제어 (RBAC)

-   모든 서비스 간 통신 TLS 1.3 적용
